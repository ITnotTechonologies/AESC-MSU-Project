## Проект: сайт доставки для СУНЦ

## 1. Общая информация

Я создаю учебный веб-проект: сайт доставки для СУНЦ.  
Цель — быстро собрать рабочий MVP за 3 дня, с возможностью дальнейшего расширения.

Проект должен быть:

- понятным;
    
- визуально аккуратным;
    
- адаптивным;
    
- простым в поддержке;
    
- реально реализуемым в короткий срок.
    

### Основная идея

Пользователь заходит на сайт, авторизуется, выбирает товары, выбирает курьера, оформляет заказ, отслеживает статус заказа и может общаться с курьером через чат.

### Целевая аудитория

- ученики СУНЦ;
    
- преподаватели;
    
- воспитатели.
    

### Особенность проекта

Есть отдельная категория товаров для доставки из:

- Ozon;
    
- Яндекс Маркета.
    

Также нужен выбор курьера и доставка в любую точку на территории СУНЦ.

## 3. Ограничения проекта

Нужно учитывать:

- срок реализации очень короткий, всего 3 дня;
    
- нужен именно MVP;
    
- приоритет — работоспособность, а не идеальная архитектура;
    
- сложные технологии использовать только если они реально оправданы;
    
- важно, чтобы код был читаемым и легко дописывался вручную;
    
- не нужно городить избыточную микросервисную архитектуру;
    
- не нужно делать слишком сложный frontend;
    
- real-time чат можно упростить до polling, если WebSockets не успеваем.

## 4. MVP-цели

Минимально жизнеспособная версия должна включать:

1. регистрацию и вход;
    
2. просмотр каталога товаров;
    
3. фильтрацию товаров по категориям;
    
4. выбор товара и добавление в корзину;
    
5. оформление заказа;
    
6. выбор курьера;
    
7. выбор точки доставки;
    
8. просмотр истории заказов;
    
9. отслеживание статуса заказа;
    
10. чат с курьером в упрощенном виде;
    
11. страницу профиля;
    
12. страницу 404;
    
13. страницу “О проекте” и контакты.

Нужно обязательно обеспечить:

- доступ к заказам только для владельца заказа, курьера и админа;

## 11. Поведение чата и статусов

Для MVP допустимо:

- хранить сообщения в БД;
    
- обновлять чат через polling каждые 5–10 секунд;
    
- обновлять статус заказа через polling;
    
- WebSockets можно оставить как улучшение.
    

Если используется polling, то нужно:

- не перегружать сервер;
    
- делать аккуратный интервал обновления;
    
- показывать время последнего обновления.
