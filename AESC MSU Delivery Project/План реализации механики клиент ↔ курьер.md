# План реализации механики клиент ↔ курьер

## Целевая логика

```
created -> accepted -> delivering -> delivered -> received
             ↘
              cancelled / returned
```

Чат при этом не является “истиной” состояния.  
Истина хранится в **status заказа в БД**, а чат только отображает системные события и сообщения людей.

---

## Шаг 1. Зафиксировать правила игры

Сначала надо очень чётко определить, что означает каждый статус и кто что может нажимать.

### Статусы

- `created` — заказ создан клиентом
    
- `accepted` — курьер принял заказ
    
- `delivering` — курьер уже несёт заказ к СУНЦ
    
- `delivered` — курьер донёс заказ до точки выдачи
    
- `received` — клиент забрал заказ
    
- `cancelled` — заказ отменён или отклонён
    

### Кто что делает

**Клиент**

- создаёт заказ
    
- пишет в чат
    
- подтверждает получение заказа
    

**Курьер**

- принимает заказ
    
- отклоняет заказ
    
- отмечает, что донёс заказ
    
- пишет в чат
    

**Система**

- добавляет уведомления в чат
    
- меняет статус заказа
    
- показывает, что произошло
    

### Что важно

Не делать действия “просто сообщением в чате”.  
Кнопки должны менять **статус**, а чат должен только показывать событие.

---

## Шаг 2. Обновить БД под эту механику

Тебе, скорее всего, понадобятся 2 маленьких улучшения в БД:

### 2.1. Расширить статусы заказа

Если сейчас в ENUM нет `received`, его нужно добавить.

### 2.2. Пометить системные сообщения

В `messages` полезно добавить:

- `message_type` = `user` / `system`
    

Или хотя бы флаг:

- `is_system BOOLEAN DEFAULT FALSE`
    

Это поможет выводить в чате:

- обычные сообщения
    
- автоматические уведомления
    

---

## Шаг 3. Продумать API-эндпоинты

Нужно сделать маленький, понятный набор эндпоинтов.

### Для курьера

- `POST /orders/{id}/accept`
    
- `POST /orders/{id}/reject`
    
- `POST /orders/{id}/mark-delivered`
    

### Для клиента

- `POST /orders/{id}/confirm-received`
    

### Для чата

- `GET /orders/{id}/messages`
    
- `POST /orders/{id}/messages`
    

### Для интерфейса

- `GET /profile` или `GET /order/{id}`
    
- `GET /courier/orders`
    
- `GET /client/orders`
    

---

## Шаг 4. Описать правила доступа

Это очень важно, чтобы никто не мог нажимать чужие кнопки.

### Курьер может:

- видеть только свои назначенные заказы
    
- принимать только свои заказы
    
- отклонять только свои заказы
    
- менять статус только своих заказов
    

### Клиент может:

- видеть только свои заказы
    
- подтверждать только свои заказы
    
- писать только в чат своего заказа
    

### Админ может:

- видеть всё
    
- при необходимости вручную править заказы
    

---

## Шаг 5. Реализовать системные сообщения

Когда меняется статус, в чат автоматически добавляется сообщение, например:

- “Курьер принял заказ”
    
- “Курьер отправился к точке доставки”
    
- “Курьер доставил заказ в СУНЦ”
    
- “Клиент подтвердил получение заказа”
    

То есть:

1. кнопка нажата;
    
2. статус в БД обновился;
    
3. в `messages` добавилось системное сообщение;
    
4. интерфейс это показал.

## Шаг 6. Подключить это к Jinja2

У тебя Jinja2, значит логика будет простой:

### На странице клиента

Показывать:

- статус заказа
    
- чат
    
- кнопку `Подтвердить получение`, если статус `delivered`
    

### На странице курьера

Показывать:

- список назначенных заказов
    
- кнопки:
    
    - `Принять`
        
    - `Отклонить`
        
    - `Доставил`
        
- чат по каждому заказу
    

### На профиле клиента

Показывать:

- историю заказов
    
- текущий статус
    
- последнее сообщение из чата
    

---

## Шаг 7. Сделать обновление без перезагрузки

Для MVP тебе хватит:

- обычного polling каждые 5–10 секунд
    

Что обновлять:

- статус заказа
    
- новые сообщения
    

Это проще, чем WebSockets, и для учебного проекта нормально.

---

## Шаг 8. Протестировать весь сценарий

Нужно пройти весь путь руками:

### Сценарий клиента

1. оформить заказ
    
2. увидеть его в профиле
    
3. дождаться принятия
    
4. увидеть, что курьер принял
    
5. увидеть, что курьер доставил
    
6. нажать подтверждение получения
    

### Сценарий курьера

1. войти как курьер
    
2. увидеть новый заказ
    
3. нажать принять или отклонить
    
4. после принятия нажать доставил
    

### Сценарий системы

1. при каждом статусе создать системное сообщение
    
2. в чате показать это уведомление

## Промпт 1. Спроектировать логику статусов и прав доступа

```text
Ты senior backend architect. У меня учебный проект на FastAPI + Jinja2 + PostgreSQL для доставки в СУНЦ.

Требования:
- заказ создаёт клиент;
- курьер видит новый заказ и может принять или отклонить;
- после принятия он нажимает кнопку "доставил";
- после доставки клиент нажимает "подтвердить получение";
- чат используется для обычных сообщений и системных уведомлений;
- важна защита: клиент видит только свои заказы, курьер — только назначенные ему.

Сделай:
1. список статусов заказа;
2. таблицу прав доступа;
3. список системных сообщений;
4. список API-эндпоинтов;
```

# Самый практичный совет

Старайся каждый раз давать нейросети **не весь проект**, а только один маленький участок:

- только БД;
    
- только один роут;
    
- только один шаблон;
    
- только polling;
    
- только тесты.
    

Так она будет попадать точнее и не будет “разносить” архитектуру.
